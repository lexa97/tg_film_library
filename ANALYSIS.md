## 1. Архитектура и структура

**Правило:** handlers — приём обновлений; services — бизнес-логика; db — репозитории (CRUD).

**Факт:**
- Структура папок совпадает с правилами.
- В `app/services/` лежит не «бизнес-логика в классах», а **функции доступа к БД** (`user_group.py`, `film.py`) и функции работы с TMDB (`tmdb.py`). То есть по смыслу часть кода в `services/` — это репозиторный слой, но без классов и без разделения «репозиторий / сервис».
- Класс `Database` в `app/db/service.py` нигде не используется: хендлеры получают `AsyncSession` из middleware и вызывают функции из `app.services.user_group` и `app.services.film` напрямую.

**Вывод:** Разделение по папкам есть, но по правилам нужны: репозитории (классы в `db/`), сервисы (классы в `services/`), хендлеры вызывают только сервисы и не работают с сессией/репозиториями напрямую.

---

## 2. Код только на классах

**Правило:** Вся бизнес-логика в классах (UserGroupService, FilmService, GroupFilmService, TMDBFilmSearch). Функции — только утилиты или регистрация роутеров.

**Факт:**
- `app/services/user_group.py` — только функции: `get_or_create_user`, `get_user_groups`, `create_group`, `add_member_to_group`, `is_group_admin` и т.д.
- `app/services/film.py` — только функции: `find_film_by_external`, `create_film`, `add_film_to_group`, `get_group_films`, `mark_as_watched` и т.д.
- `app/services/tmdb.py` — только функции: `search_multi`, `fetch_movie_or_tv`, `result_to_film_data`, `format_poster_url`.

**Вывод:** Правило «код только на классах» не выполнено. Нужны классы вида `UserGroupService`, `FilmService` / `GroupFilmService`, `TMDBFilmSearch` (или аналог) с методами вместо модульных функций.

---

## 3. Наследование и абстракция поисковика

**Правило:** Базовые/абстрактные классы где уместно; для поиска фильмов — интерфейс (BaseFilmSearchProvider) и реализация (TMDBFilmSearch).

**Факт:**
- Нет `BaseFilmSearchProvider` (ABC) и класса `TMDBFilmSearch`.
- Нет `BaseRepository`.

**Вывод:** Абстракция поисковика и репозиториев отсутствует. Для расширения (например, второй источник фильмов) по правилам нужен интерфейс и классы.

---

## 4. Бизнес-логика в сервисах

**Правило:** Решения «добавить в группу», «можно ли отметить просмотренным», сценарии «подтвердить фильм» и т.д. — в сервисах, оформлены в отдельные методы. Хендлер только вызывает метод сервиса.

**Факт:**
- **group.py, on_contact:** хендлер сам проверяет «есть ли группа», `is_group_admin`, «есть ли user_id в контакте», `get_user_by_telegram_id`, `add_member_to_group`, затем сам шлёт уведомление новому пользователю. Это сценарий «добавить участника по контакту», его по правилам должен выполнять один метод сервиса (например, `add_member_by_contact_for_current_admin(contact)`) с уведомлениями внутри сервиса или отдельного сервиса.
- **film_search.py, on_confirm_film:** хендлер вызывает `fetch_movie_or_tv`, `find_film_by_external`, `create_film`, `get_or_create_user`, `add_film_to_group`, `get_group_member_telegram_ids` и сам рассылает уведомления участникам. Это сценарий «подтвердить фильм и добавить в группу» — должен быть один метод сервиса (например, `confirm_and_add_film_to_group(...)`), уведомления — внутри сервиса.
- **list_watched.py, cb_watched:** хендлер вызывает `get_group_film_by_id`, `get_or_create_user`, `mark_as_watched`, `get_group_member_telegram_ids` и сам рассылает уведомления. Метод типа `mark_watched(group_film_id, telegram_user_id)` в сервисе должен инкапсулировать проверки, запись в БД и уведомления.

**Вывод:** Бизнес-логика и сценарии частично перенесены в хендлеры. Нужно вынести сценарии в сервисы (один вызов из хендлера = один метод сервиса, внутри — работа с репозиториями и уведомления).

---

## 5. Слой хендлеров (Telegram)

**Правило:** Хендлер: извлечь данные из message/callback → (при необходимости валидация) → вызов метода сервиса → формирование ответа (сообщение + клавиатура). В хендлерах запрещены: прямая работа с БД, бизнес-решения, парсинг ответов TMDB.

**Факт:**
- Хендлеры принимают `AsyncSession` (MagicData) и вызывают по 5–10 функций из `user_group` и `film` подряд — по сути собирают сценарий в хендлере.
- В `film_search.py`, `on_text_search`: после `search_multi` хендлер вызывает `result_to_film_data(item)` и сам формирует текст из `data["title"]`, `data.get("year")`, `data.get("description")`, `data.get("poster_url")` — это работа с DTO TMDB в хендлере. По правилам маппинг и формирование «что показать пользователю» должны быть в services.
- Прямых запросов к БД (SQLAlchemy) в хендлерах нет, но они напрямую зависят от множества функций доступа к данным и от session.

**Вывод:** Хендлеры перегружены: и сценарии, и работа с данными TMDB. Нужно: хендлер только извлекает данные из события → вызывает один метод сервиса → по возвращённому результату формирует сообщение и клавиатуру (данные для ответа должны приходить из сервиса в виде моделей/структур, не сырых dict из TMDB).

---

## 6. Pydantic

**Правило:** Все структурированные входы/выходы между слоями — Pydantic-модели. Запрещено передавать `dict` между слоями.

**Факт:**
- `app/db/service.py`: `create_film(self, film_data: dict) -> Film` — приём `dict`.
- `app/services/film.py`: `create_film(session, film_data: dict)` — приём `dict`.
- В `film_search.py` между TMDB и хендлером передаются словари (`result_to_film_data` возвращает dict, хендлер с ним работает).
- Отдельных Pydantic-моделей для запросов/ответов между хендлером и сервисом нет (кроме конфига).

**Вывод:** Правило про Pydantic не выполнено. Нужны модели типа `FilmCreate`, `FilmSearchResult`, ответы сервисов для хендлера — тоже типизированные (Pydantic или dataclass), не `dict`.

---

## 7. Работа с БД

**Правило:** Доступ к БД только через репозитории. Один репозиторий — одна сущность (users, groups, films, group_films, watched). Репозитории без бизнес-логики.

**Факт:**
- Отдельных классов-репозиториев нет. CRUD выполняется функциями в `app/services/user_group.py` (User, Group, GroupMember) и `app/services/film.py` (Film, GroupFilm, Watched).
- Класс `Database` в `db/service.py` — фасад над этими же функциями, но **не используется** в приложении.

**Вывод:** Репозитории по правилам не выделены. Имеет смысл: либо вынести CRUD в классы в `db/` (UserRepository, GroupRepository, FilmRepository, GroupFilmRepository, WatchedRepository), либо явно оформить текущие функции как репозиторный слой и дать им имена/папку по правилам, а бизнес-сценарии перенести в сервисы, которые эти функции вызывают.

---

## 8. Сервисы и TMDB

**Правило:** Вся работа с TMDB в services; абстракция «поисковик» (интерфейс + реализация); маппинг ответов TMDB только в services; лимиты и при необходимости кэш.

**Факт:**
- Вызовы TMDB делаются из `app/services/tmdb.py` — по месту ок.
- Абстракции поисковика нет — только функции.
- Маппинг `result_to_film_data` вызывается из хендлера в `film_search.py` — нарушение.
- Лимиты запросов TMDB и кэш не реализованы (можно оставить как рекомендацию).

**Вывод:** Нужны: класс-провайдер поиска (например, TMDBFilmSearch) с методом поиска и маппингом в DTO; хендлер не должен вызывать `result_to_film_data` и не должен работать с dict из TMDB.

---

## 9. Пользователи и группы

**Правило:** Группа — логическая сущность в боте; добавление по контакту через `contact.user_id`; MVP — одна группа на пользователя; проверка прав в сервисах.

**Факт:**
- Используется `getattr(contact, "user_id", None)` — по смыслу то же, что `contact.user_id`.
- Одна группа на пользователя в сценариях соблюдается.
- Проверка «админ ли» сейчас в хендлере (вызов `is_group_admin`), по правилам проверка прав должна быть внутри сервиса.

**Вывод:** Поведение по ТЗ в целом соблюдено; для полного соответствия правилам проверку прав и сценарий добавления по контакту лучше перенести в сервис.

---

## 10. Дополнительные замечания

- **Дублирование импорта:** в `film_search.py` внутри хендлера есть `from app.services.tmdb import fetch_movie_or_tv` и `from app.services.user_group import get_or_create_user` — уже импортированы в начале файла. Убрать дубли.
- **get_settings():** при каждом вызове создаётся новый экземпляр `Settings()`. Для конфига обычно делают кэш/синглтон, чтобы не читать .env многократно (не критично, но полезно).
- **utils/:** в правилах указана папка `utils/`, в проекте она может быть пустой — допустимо для MVP.

## Рекомендуемый порядок доработок

1. **Pydantic-модели:** ввести DTO для обмена между слоями (FilmCreate, FilmSearchResult, ответы для списка/карточки). Убрать передачу `dict` в `create_film` и из TMDB в хендлер.
2. **Репозитории:** оформить доступ к БД в классах в `db/` (по одной сущности: User, Group, GroupMember, Film, GroupFilm, Watched), без бизнес-логики.
3. **Сервисы (классы):** ввести классы UserGroupService, FilmService/GroupFilmService с методами сценариев (add_member_by_contact_for_current_admin, confirm_and_add_film_to_group, mark_watched с уведомлениями и т.д.), использующие репозитории и (при необходимости) сервис уведомлений.
4. **Поиск фильмов:** ввести BaseFilmSearchProvider (ABC) и TMDBFilmSearch в `services/`, маппинг TMDB → DTO только внутри сервиса/провайдера; хендлер получает уже готовые DTO.
5. **Хендлеры:** переписать так, чтобы хендлер только извлекал данные из message/callback, вызывал один метод сервиса и по возвращённому результату формировал ответ (сообщение + клавиатура). Убрать из хендлеров работу с session и прямые вызовы репозиториев/функций БД; сессию в сервисы передавать через фасад или DI (например, через тот же класс Database или внедрение репозиториев в сервисы при старте).
6. **Database в db/service.py:** либо начать использовать его в хендлерах (хендлеры вызывают только `db.*`), либо удалить, если решите делать DI сервисов с репозиториями без фасада.

После этих шагов код будет соответствовать правилам из rules.md и .cursorrules.
