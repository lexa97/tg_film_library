# ТЗ: команда `/relative` и кэш рекомендаций TMDB

## Цель

Выдавать пользователю **до 5** рекомендованных фильмов/сериалов на основе **просмотренных** в его группе: данные берутся из **локального кэша** (результаты TMDB `recommendations` по каждому исходному фильму), без массовых запросов к API в момент команды.

Отображение и действия — **как при текстовом поиске**: карточка (постер, заголовок, год, описание, тип), кнопки **«Подтвердить»** и **«Скачать»** (`build_film_confirm_keyboard` в `app/keyboards/inline.py`), тот же callback `confirm_film:...` (обработчик в `app/handlers/film.py`).

Абстракция провайдера — расширение контракта в духе `app/services/film_search.py` (`BaseFilmSearchProvider`).

---

## Предпосылки по данным

1. **`Film.media_type`** (`movie` | `tv`) должен **храниться в БД** и заполняться при добавлении фильма (сейчас в `FilmCreate` / репозитории поле есть, в модели `Film` при необходимости добавить колонку и `initdb`/миграцию).
2. Связь кэша с каталогом — через **`film_id` (PK `films`)**, а не только `external_id`: у TMDB числовой id у фильма и сериала — разные сущности при одном номере.

---

## Модель БД (кэш)

**Таблица** `film_recommendation_cache` (имя можно уточнить, смысл — строка = одна рекомендация TMDB для одного нашего фильма-источника).

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | PK | Автоинкремент |
| `source_film_id` | FK → `films.id`, индекс | Фильм из нашей базы, для которого тянули рекомендации |
| `recommended_external_id` | string | TMDB id рекомендованного title |
| `recommended_media_type` | string | `movie` или `tv` |
| `fetched_at` | datetime | Когда строка записана/обновлена при последнем обходе |

**Ограничения:**

- `UNIQUE (source_film_id, recommended_external_id, recommended_media_type)` — без дублей в кэше.
- При полном обновлении кэша для данного `source_film_id`: удалить старые строки по этому ключу и вставить новый набор **или** upsert с последующим удалением «устаревших» (проще: delete + bulk insert для источника).

**Объём TMDB:** для каждого источника запрашивать **первую страницу** `GET /3/movie/{id}/recommendations` или `GET /3/tv/{id}/recommendations` (язык как у проекта, например `ru-RU`). Дополнительно пагинацию в ТЗ не требовать, чтобы не раздувать фон.

**Репозиторий:** только CRUD по этой таблице, без бизнес-логики (`rules.md`).

---

## Провайдер TMDB

В **`app/services/tmdb_provider.py`** (единый провайдер с `fetch_film`, как в контейнере):

- Новый метод, например `fetch_recommendations(external_id: str, media_type: str) -> list[tuple[str, str]]` или список минимальных DTO с `external_id` + `media_type`.
- Реализация: HTTP к `/{movie|tv}/{id}/recommendations`, разбор `results`: у каждого элемента взять `id` и зафиксировать тип (`movie` / `tv` — в ответе рекомендаций обычно тот же тип, что и источник; зафиксировать в коде явно).

В **`BaseFilmSearchProvider`** (`film_search.py`) объявить абстрактный метод с той же семантикой, чтобы не ходить в TMDB из хендлеров.

## Фоновое обновление кэша

**Периодичность:** не чаще **раз в сутки** (константа или `pydantic-settings`, например `RECOMMENDATIONS_CACHE_INTERVAL_HOURS=24`).

**Набор источников:** все фильмы `films`, у которых есть хотя бы одна запись в `group_films` (активный каталог групп). Альтернатива с меньшей нагрузкой — только фильмы, встречающиеся в `watched`; в ТЗ зафиксировать один вариант (рекомендуется **все из списков групп**, чтобы кэш был полезен и для «ещё не просмотренных» источников при смене логики).

**Алгоритм одного прогона:**

1. Выбрать уникальные `film_id` из `group_films` (или согласованный критерий).
2. Для каждого `Film`: взять `external_id`, `media_type`, `source` (только `tmdb`; иное — пропуск).
3. Вызвать `fetch_recommendations`; при ошибке сети/API — залогировать, для этого источника кэш не ломать или не обновлять (на усмотрение: проще **не удалять старый кэш** при ошибке).
4. Транзакционно для источника: удалить старые строки `film_recommendation_cache` по `source_film_id`, вставить новые.
5. Учесть **rate limit TMDB**: небольшая пауза между запросами или ограничение параллелизма (последовательно в MVP — приемлемо).

**Способ запуска в процессе бота (на выбор реализации, указать в PR):**

- `asyncio` + `asyncio.sleep` цикл в фоновой задаче, стартующей из `main.py` после `start_polling`, с учётом интервала; **или**
- отдельный скрипт `python -m app.jobs.refresh_recommendation_cache` и вызов по cron в Docker.

**Логирование:** INFO — сколько источников обработано, WARNING/ERROR — сбои по TMDB.

## Сервис (бизнес-логика)

Новый класс, например **`RecommendationService`** в `app/services/`:

1. **`build_relative_candidates(session, group_id, limit=5)`** (или аналогичное имя):
   - Определить группу пользователя (как в остальных сценариях).
   - Взять **просмотренные** в этой группе: `group_films` + join `watched` + `films`.
   - Если просмотренных нет — вернуть понятную ошибку/пустой результат (**не** подмешивать весь список без согласования; при желании позже расширить ТЗ).
   - Для каждого просмотренного `film_id` прочитать строки из `film_recommendation_cache`.
   - Собрать пары `(recommended_external_id, recommended_media_type)` → **число вхождений** (сколько раз title всплыл из разных источников).
   - Исключить title, которые **уже есть в списке группы** (join `group_films` → `films` по `external_id` + `media_type`).
   - Отсортировать по убыванию счётчика; при равенстве — стабильный вторичный ключ (например лексикографически по `external_id` или порядок первого появления).
   - Взять первые `limit`.
2. Для отображения каждого кандидата вызвать существующий **`fetch_film`** провайдера; из ответа собрать **`FilmSearchResult`** (`app/services/dto.py`) — те же поля, что ожидает `build_film_confirm_keyboard` и разметка в `film.py` (в т.ч. `media_type`).

Ошибки TMDB при `fetch_film` для одного из топ-5: пропустить кандидата и при необходимости добрать следующий по рангу или уменьшить число карточек (задокументировать поведение: «до 5, меньше если часть не загрузилась»).

## Команда `/relative`

**Регистрация:** `BotCommand` в `app/main.py` рядом с `/start` и `/list`.

**Хендлер:** в `app/handlers/commands.py` или отдельный роутер, подключённый **до** общего текстового хендлера в `film.py`.

**Поведение:**

- Только для пользователя в группе; иначе — то же сообщение, что и для других команд.
- Вызов `RecommendationService` → список `FilmSearchResult`.
- Если кэш пуст для всех просмотренных: сообщение вида «Кэш ещё не готов, попробуйте позже» (после первого ночного прогона — не должно повторяться).
- Если после фильтрации кандидатов 0: «Пока нечего рекомендовать (все уже в списке или мало данных)».
- Успех: цикл отправки **как в `search_film`** в `film.py`: заголовок «Найдено…» / для relative — свой текст («Подборка по просмотренным»), затем для каждого результата — тот же HTML, постер, `build_film_confirm_keyboard(result, index)`.

Повторное использование: вынести общую функцию **«отправить список карточек FilmSearchResult»** в хелпер (например `app/handlers/film_cards.py` или метод утилиты), чтобы не дублировать 40 строк; в ТЗ это пожелание, не блокер.
